# Archive and Continuation Policy — v3.1

## Purpose

This policy prevents the repository from growing forever without focus.

---

# 1. Repository boundary

This repository is a business and investor dossier.

It should not become:

```text
firmware lab
website codebase
raw supplier database
random idea archive
general AndyAI notebook
```

---

# 2. Allowed updates after v3.1

Allowed:

```text
minor corrections
broken link fixes
updated contact info
updated pricing assumptions
updated external references
links to continuation repos
new release note only if strategically necessary
```

---

# 3. Not allowed after v3.1

Avoid:

```text
large new conceptual expansions
unrelated ideas
firmware source code
landing page source code
unvalidated claims
unstructured brainstorming
```

---

# 4. How to continue properly

Use continuation repositories:

```text
technical prototype → andyai-director-edge-node
landing page → espandyai-edge-landing or existing web repo
PDF exports → investor artifact pack
pilot execution → pilot tracker / CRM
```

---

# 5. Update policy

If the business dossier must be updated, use restrained patch versions:

```text
v3.1.1 = typo/legal wording correction
v3.1.2 = link/update correction
v3.2 only if major external presentation revision is required
```

Do not continue automatic large expansions unless the business purpose is clearly new.

---

# 6. Canonical stop rule

```text
When a repo has answered its mission, stop expanding it and create the next focused repo.
```

This rule is part of the AndyAI TAPFORGE discipline.

Factory power requires boundaries.

Without boundaries, even a powerful factory becomes fog.

---

# 7. Final doctrine

```text
Plan here.
Build elsewhere.
Export cleanly.
Pitch clearly.
Pilot concretely.
```
